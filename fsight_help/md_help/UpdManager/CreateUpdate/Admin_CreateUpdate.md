# Создание обновления


Для создания обновления войдите в репозиторий под одной из следующих
 учётных записей:


	- администратор, отвечающий за синхронизацию объектов между репозиториями
	 или синхронизацию версий прикладных систем у разработчиков и у конечных
	 пользователей продукта «Форсайт. Аналитическая платформа»;


	- член группы «[Администраторы](Admin.chm::/03_Admin/Groups/Admin_Groups.htm)»;


	- пользователь/член группы, обладающий привилегией «[Доступ
	 к менеджеру обновлений](admin.chm::/04_SecurityPolicy/Admin_Priv.htm#upd_manager)».


При работе над прикладными системами с использованием «Форсайт. Аналитическая платформа»
 для разделения процессов разработки, как правило, создаются несколько
 репозиториев - разработка, тестирование, эксплуатация. В процессе разработки
 проекта часто происходит изменение объектов прикладной системы, а также
 модификация состава системы (добавление или удаление объектов).


С помощью обновления можно внести изменения в клонированный репозиторий.
 Клонированный репозиторий создается посредством сохраненной копии. Клон
 репозитория создаётся средствами соответствующей СУБД.


Пример процесса обновления в два этапа:


	- Создание обновления в репозитории разработчиков.


	- Проведение обновления в клонированном репозитории
	 для тестирования.


Примечание.
 Если в репозитории заказчика обновляемая прикладная система интенсивно
 используется, то рекомендуется выделить время на остановку системы для
 обновления, либо производить обновление в момент наименьших обращений
 пользователей. При обновлении систем на основе веб-приложения рекомендуется
 предварительно останавливать сайт с приложением.


Обновление объектов репозитория представляет собой файл с расширением
 *.pefx (Platform Export File Extended), который содержит различную информацию
 по изменению объектов прикладных систем.


Каждому объекту репозитория соответствует уникальный цифровой код, который
 идентифицирует объект. Эти коды хранятся в системных таблицах, которые
 также хранят полную информацию об объектах и связях между ними. В клонированных
 репозиториях и в первоначальном репозитории уникальные коды у одних и
 тех же объектов совпадают. Если репозиторий не является клонированным,
 то внутренние коды у объектов обновления и объектов репозитория могут
 при обновлении не совпадать, а при добавлении и пересекаться, что приведет
 к возникновению конфликтов.


Также в процессе установки обновления следует учитывать проверку версии
 объектов при установленном [контроле
 версий объектов при обновлении](Admin.chm::/04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm#version_control).


Примечание.
 Обновление версий объектов рекомендуется производить только для клонированных
 репозиториев.


Основные причины, по которым нельзя производить обновление для разных
 репозиториев:


	- совпадение внутренних кодов у объектов;


	- совпадение идентификаторов у объектов;


	- защита от создания новых объектов с внутренними кодами,
	 которые уже существуют.


Исключением может быть случай, когда необходимо перенести объекты в
 пустой репозиторий.


## Создание обновления


Для создания обновления выполните шаги:


	- Создайте или добавьте [папки](Admin_CreateUpdate_CreateFolder.htm)
	 для упорядочивания структуры обновления.


Примечание.
 Создание и добавление папок из менеджера обновлений доступно только в
 настольном приложении.


	- Добавьте [объекты
	 репозитория](Admin_CreateUpdate_AddObjects.htm).


	- Проведите [анализ
	 изменений](../Admin_UpMBObj_ChangeAnalysis.htm) объектов репозитория и добавьте в обновление при необходимости.


Примечание.
 Проведение анализа изменений объектов репозитория доступно только в настольном
 приложении.


	- Добавьте [SQL-оператор](Admin_CreateUpdate_AddSQL.htm).


	- Добавьте [удаление объекта](Admin_CreateUpdate_AddDel.htm).


	- Добавьте [комментарий
	 к обновлению](Admin_CreateUpdate_AddComent.htm).


Примечание.
 Добавление комментария к обновлению доступно только в настольном приложении.


	- Добавьте [субъект безопасности](Admin_CreateUpdate_AddUser.htm).


Совет.
 При добавлении в обновление [доменных
 пользователей](Admin.chm::/03_Admin/Users/Admin_UserCreate_Domain.htm) или [групп](admin.chm::/03_Admin/Groups/Admin_GroupsCreateDomain.htm)
 рекомендуется проверять идентичность [настроек LDAP](Admin.chm::/03_Admin/communication_with_ad.htm)
 в файлах [settings.xml](setup.chm::/UiWebSetup/03_Setup_Web/BI_Server_Settings_XML.htm),
 расположенных в различных окружениях. Если настройки совпадают, то для
 корректной работы с доменными пользователями или группами переносите их
 в рамках одного приложения, в котором они были подключены: либо в веб-приложении,
 либо в настольном приложении.


	- Добавьте [специальные
	 объекты](Admin_CreateUpdate_SpecialObjects.htm).


Примечание.
 Добавление специальных объектов доступно только в настольном приложении.


	- Добавьте [права доступа](AddAccessPermissions.htm).


	- Настройте параметры обновления [выбранных
	 объектов](../Admin_UpMBObj_OBJUpdateParameters.htm) и [процесса
	 обновления](../UpdateProcess_Parameters.htm).


	- Используйте [фильтрацию объектов](Filtering.htm)
	 для удобства работы со структурой обновления.


Примечание.
 Использование фильтрации объектов доступно только в настольном приложении.


	- Задайте [порядок
	 объектов](../Admin_UpMBObj_ObjView.htm#order) и настройте их [отображение](../Admin_UpMBObj_ObjView.htm)
	 при необходимости.


	- [Удалите объект](Admin_CreateUpdate_DelObjects.htm)
	 из обновления при необходимости.


	- Установите [флаги обновления](../Admin_UpdateFlags.htm)
	 при необходимости.


Примечание.
 Использование флагов обновления доступно только в настольном приложении.


	- Рассчитайте [контрольные
	 суммы](Admin.chm::/04_SecurityPolicy/Integrity_control.htm#metadata) для дальнейшей проверки целостности метаданных
	 при необходимости.


	- [Сохраните обновление](../SaveUpdate/Admin_SaveUpdate.htm).

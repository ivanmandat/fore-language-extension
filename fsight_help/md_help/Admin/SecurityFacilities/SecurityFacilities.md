# Руководство по комплексу средств защиты


	Модель безопасности определяет комплекс средств, обеспечивающих
	 защиту от несанкционированного доступа к прикладным системам, методы
	 разграничения доступа субъектов к объектам прикладных систем, принципы
	 аутентификации субъектов, аудита их действий в системе, контроля целостности
	 программной системы и хранящейся в ней информации.


## Генерация комплекса средств защиты


	Для генерации комплекса средств защиты:


		- Установите и настройте компоненты, входящие в программный
		 продукт «Форсайт. Аналитическая платформа»:


			- [настольное
			 приложение](Setup.chm::/02_AK_Install/Setup_AK_Install.htm);


			- [BI-сервер](Setup.chm::/UiWebSetup/UiWebSetup_TitlePage.htm);


			- [сервер
			 веб-приложения](Setup.chm::/UiWebSetup/03_Setup_Web/Web_server_setup.htm);


			- сервер лицензий на базе [Guardant](Setup.chm::/08_Licensing/Guardant/License_Server_Guardant.htm)
			 или [RMS](Setup.chm::/08_Licensing/RMS/LicenseServer.htm);


			- [сервер
			 баз данных](Setup.chm::/03_DB_Server_Config/Setup_DB_Server_Config.htm);


			- [сервер
			 приложений](Setup.chm::/UiWebSetup/AppServer.htm);


			- [сервер
			 состояний](Setup.chm::/UiWebSetup/01_BIServer/StateServer/State_Server.htm). В качестве сервера состояний
			 платформа поддерживает работу с [Redis](Setup.chm::/UiWebSetup/01_BIServer/StateServer/Redis.htm)
			 и [Memcached](Setup.chm::/UiWebSetup/01_BIServer/StateServer/Memcached.htm).
			 Сервер состояний не входит в комплект поставки продукта «Форсайт. Аналитическая платформа».


	«Форсайт. Аналитическая платформа»
	 может работать в различных [конфигурациях](Setup.chm::/GeneralInfo.htm),
	 используя отдельные серверы для выделения ядра бизнес-логики или пользовательского
	 интерфейса. Для повышения отказоустойчивости в каждой конфигурации
	 может быть реализован [кластер
	 BI-серверов](Setup.chm::/UiWebSetup/01_BIServer/Cluster/Cluster_Of_BI_Server.htm). При использовании двух и более BI-серверов
	 рекомендуется добавлять в конфигурацию сервер состояний, с помощью
	 которого будут восстанавливаться параметры объектов в случае сбоя
	 BI-сервера, к которому выполняется запрос. Для получения подробной
	 информации о конфигурациях систем и принципах их работы обратитесь
	 к разделу «[Конфигурации
	 систем и принцип их работы](Setup.chm::/GeneralInfo.htm)».

	 для конечных пользователей необходимо использовать дистрибутив FP10Standard.exe, который не содержит
	 средств разработки и администрирования. Это необходимо для повышения
	 безопасности системы. Для администраторов и разработчиков системы
	 используйте дистрибутив FP10Full.exe
	 (полный комплект настольного приложения) из [комплекта поставки](Setup.chm::/02_AK_Install/DistributionKit.htm).
	 Данный дистрибутив содержит все компоненты, в том числе средства разработки
	 и администрирования, а также позволяет выборочно определять устанавливаемые
	 компоненты.

	 обратитесь к разделу «[Установка продукта](Setup.chm::/Install_Title.htm)».

Управление режимом обслуживания.


	При отключенном разделении ролей между администратором информационной

	 встроенной группе пользователей «Администраторы», а также пользователям,
	 имеющим соответствующие привилегии.


	При разделении ролей между администратором информационной безопасности
	 (АИБ) и прикладным администратором (ПА) – АИБ может осуществлять вход
	 только в менеджер безопасности и имеет доступ к разделам:


		- навигатор;


		- пользователи;


		- группы;


		- привилегии;


		- атрибутный доступ;


		- мандатный доступ;


		- редактор политик;


		- аудит;


		- протокол доступа;


		- мониторинг нарушений защиты;


		- сервис.


	Прикладной администратор имеет доступ к разделам:


		- пользователи;


		- группы;


		- протокол доступа.


## Описание контролируемых функций


	Управление доступом:


		- идентифицирует и проверяет подлинность субъектов безопасности
		 при входе в систему по идентификатору и паролю; при [двухфакторной
		 аутентификации](Setup.chm::/UiWebSetup/Authentication/double_auth.htm) - по идентификатору, паролю
		 и сертификату;


		- идентифицирует рабочие станции, узлы компьютерной сети,
		 внешние устройства ЭВМ по их логическим адресам (номерам);


		- проверяет информацию о сроке действия учётной записи пользователя,
		 о сроке действия пароля, о блокировке пользователя, о количестве
		 неудачных попыток ввода пароля до блокировки учётной записи пользователя
		 и т.д.


	Для получения подробной информации обратитесь
	 к разделу «[Настройка
	 проверки паролей пользователей](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General_Parole.htm)»;


		- поддерживает управление доступа к объектам на основе методов:
		 [дискреционного](../04_SecurityPolicy/Admin_PermSep_D.htm),
		 [мандатного](../04_SecurityPolicy/Admin_PermSep_M.htm),
		 [атрибутного](../04_SecurityPolicy/Admin_ABAC.htm),
		 [разграничения
		 доступа по уровням](../04_SecurityPolicy/Security_levels.htm).


	Для получения подробной информации обратитесь
	 к разделу «[Выбор
	 методов разграничения доступа и их настройка](../04_SecurityPolicy/Admin_SecPolicy.htm)».


	Протоколируемые административные операции:


		- регистрирация входа (выхода) субъектов доступа в систему
		 (из системы);


		- добавление, удаление, изменение групп (изменение наименования,
		 включение пользователей в группу, и пр.);


		- добавление, удаление, изменение пользователей (изменение
		 наименования, смена пароля, и пр.);


		- выдача или отзыв (аннулирование) привилегий;


		- изменение мандатной политики: удаление/добавление/изменение
		 категорий доступа, удаление/добавление/изменение уровней категорий
		 доступа;


		- регистрация изменений политики безопасности: включение/отключение
		 АИБа, включение/отключение дискреционного/мандатного контроля
		 доступа, изменение в парольной политике, объекта по умолчанию;


		- изменение прав доступа с указанием информации о предыдущих
		 и назначенных правах (были/стали). Также может содержаться информация
		 об изменении состояния флажка «[Наследовать
		 разрешения от родительского объекта](../03_Admin/Admin_AdminObjects_Discretionary.htm)»;


		- изменение аудита объектов;


		- информация о разрешении или запрете печати на устройстве,
		 изменении уровня доступа для принтера;


		- добавление, удаление, очистка списка доступных для печати
		 принтеров;


		- добавление принтера субъекту безопасности;


		- путь и имя файла при экспорте информации;


		- принудительное отключение пользователя от репозитория, выполненное
		 администратором;


		- соединения с базами данных, которые были открыты во время
		 обновления. Фиксируется только в настольном приложении;


		- дублирование всей информации об операции при сбояха записи
		 в резервный файл.


	Мониторинг нарушений защиты:


		- сигнализирует о попытках нарушения защиты.


	Для контроля целостности платформа:


		- производит проверку контрольных сумм для исполняемых файлов
		 в момент их загрузки. Если настольное или веб-приложение установлено
		 в ОС Windows, то при каждом запуске «Форсайт. Аналитическая платформа»
		 загружается минимальный набор исполняемых файлов, а по мере обращения
		 к функциональности загружаются дополнительные библиотеки, необходимые
		 для работы. В ОС Linux проверка контрольных сумм выполняется средствами
		 операционной системы. Настройте проверку контрольных сумм в соответствии
		 с документацией на ОС Linux;


		- администратор информационной безопасности вручную проверяет
		 [целостность
		 метаданных репозитория](../04_SecurityPolicy/Integrity_control.htm) с помощью расчёта [контрольных
		 сумм](../04_SecurityPolicy/Integrity_control.htm#performing_checksum), которые вычисляются определенным образом по содержимому
		 объекта.


## Описание средств надежного восстановления


	Средства надежного восстановления:


		- Резервное копирование и восстановление политики безопасности.


	Сохранение политики безопасности, настроек
	 аудита, информации о группах и пользователях, о правах доступа на
	 объекты осуществляется путём [резервного
	 копирования и восстановления политики безопасности](../04_SecurityPolicy/Admin_Service.htm).


		- Создание и восстановление резервной копии репозитория метаданных
		 производится средствами СУБД:


			- создание резервной копии репозитория метаданных может
			 понадобиться для восстановления потерянных или измененных
			 данных;


			- восстановление репозитория из резервной копии. Восстановление
			 комплекса производится из резервной копии, созданной ранее.


## Описание старта средств вычислительной техники


	Для запуска настольного приложения продукта «Форсайт. Аналитическая платформа»
	 выполните команду «Программы >
	 «Форсайт. Аналитическая платформа»
	 > «Форсайт. Аналитическая платформа»,
	 расположенную в меню Windows.


	Для запуска веб-приложения продукта «Форсайт. Аналитическая платформа»
	 в ОС Windows и ОС Linux откройте поддерживаемый браузер и в строке
	 подключения укажите адрес веб-приложения.


### Описание процедур правильности проверки старта


	Для проверки целостности исполняемых файлов при каждом запуске продукта
	 «Форсайт. Аналитическая платформа»

				 паролей);

				 днях);


				- Установить минимальный срок действия пароля (в днях).

			 УПД.2 Реализация необходимых методов управления
			 доступом и правил разграничения доступа
			 В платформе для управления доступом субъектов доступа
			 к объектам доступа должны быть реализованы методы управления
			 доступом. Методы управления доступом реализуются в зависимости
			 от особенностей функционирования информационной системы, с
			 учетом угроз безопасности информации и должны включать один
			 или комбинацию следующих методов:


				- дискреционный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе идентификационной информации субъекта и для
				 каждого объекта доступа – списка, содержащего набор субъектов
				 доступа (групп субъектов) и ассоциированных с ними типов
				 доступа;


				- мандатный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе сопоставления классификационных меток каждого
				 субъекта доступа и каждого объекта доступа, отражающих
				 классификационные уровни субъектов доступа и объектов
				 доступа, являющиеся комбинациями иерархических и неиерархических
				 категорий;


				- метод разграничения доступа по уровням осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе сопоставления меток критичности каждого субъекта
				 доступа и каждого объекта доступа, отражающих иерархический
				 уровень безопасности;


				- атрибутный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе политик и правил, которые содержат набор условий
				 для проверки атрибутов субъектов и объектов/сегментов
				 данных.

				 метода обратитесь к разделу «[Настройка
				 дискреционного метода](../04_SecurityPolicy/Admin_PermSep_D.htm)»;


				- для разграничения доступа с помощью мандатного метода

				 обратитесь к разделу «[Настройка
				 метода разграничения доступа по уровням](../04_SecurityPolicy/Security_levels.htm)»;


				- для разграничения доступа с помощью атрибутного
				 метода обратитесь к разделу «[Настройка
				 атрибутного метода](../04_SecurityPolicy/Admin_ABAC.htm)».


			Комбинация методов разграничения доступа выполняется в редакторе
			 политик менеджера безопасности на вкладке «[Контроль
			 доступа](../04_SecurityPolicy/Admin_SecPolicy.htm#access_control)». Для получения подробной информации обратитесь
			 к разделу «[Выбор
			 методов разграничения доступа и их настройка](../04_SecurityPolicy/Admin_SecPolicy.htm)».


			 УПД.4 Разделение полномочий пользователей,
			 администраторов и лиц, обеспечивающих функционирование информационной
			 системы
			 Платформа должна обеспечивать техническую возможность
			 разделения полномочий (ролей) пользователей, администраторов
			 и лиц, обеспечивающих функционирование информационной системы,
			 и санкционирования доступа к объектам доступа в соответствии
			 с разделением полномочий (ролей).


			Доступ к объектам доступа с учетом разделения полномочий

			 РСБ.4 Реагирование на сбой при регистрации
			 событий безопасности
			 В информационной системе должно осуществляться реагирование
			 на сбои при регистрации событий безопасности, в том числе
			 аппаратные и программные ошибки, сбои в механизмах сбора информации
			 и достижение предела или переполнения объема (емкости) памяти.


			Реагирование на сбои при регистрации событий безопасности
			 должно предусматривать:


				- предупреждение (сигнализация, индикация) администраторов
				 о сбоях (аппаратных и программных ошибках, сбоях в механизмах
				 сбора информации или переполнения объема (емкости) памяти)
				 при регистрации событий безопасности;


				- реагирование на сбои при регистрации событий безопасности
				 путем изменения администраторами параметров сбора, записи
				 и хранения информации о событиях безопасности, в том числе
				 отключение записи информации о событиях безопасности от
				 части компонентов информационной системы, запись поверх
				 устаревших хранимых записей событий безопасности.


			Правила и процедуры реагирования на сбои при регистрации
			 событий безопасности регламентируются в организационно-распорядительных
			 документах оператора по защите информации.


			 В менеджере безопасности для предупреждения (сигнализация,
			 индикация) администраторов о сбоях при регистрации событий
			 безопасности используются пороговые значения в разделе «[Редактор политик](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm)». Установите
			 флажок «[Пороговые
			 значения объема протокола доступа](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm#threshold_values)» на вкладке «[Общая
			 политика](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm)» и задайте первое и второе пороговые значения
			 (число записей). При каждом входе в менеджер безопасности
			 под пользователем, имеющим привилегию «[Очистка
			 протокола доступа](../04_SecurityPolicy/Admin_Priv.htm)», проверяется, достиг ли объем протокола
			 доступа пороговых значений. В случае, если было достигнуто
			 пороговое значение, то будет предложено очистить протокол
			 доступа. Для получения подробной информации обратитесь к разделу
			 «[Общие
			 настройки политики](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm)».


			Для протоколирования административных операций настройте
			 объем ведения аудита в разделе «[Классы
			 объектов](../04_SecurityPolicy/Admin_Object_Classes.htm)». Для получения подробной информации обратитесь
			 к разделу «[Настройка
			 аудита действий пользователя](../04_SecurityPolicy/Admin_Object_Classes.htm)».


			 РСБ.5 Мониторинг результатов регистрации событий

			 в соответствии с РСБ.1, и с периодичностью, установленной

			 РСБ.7 Защита информации о событиях безопасности

			 определенными [привилегиями](../04_SecurityPolicy/Admin_Priv.htm)
			 пользователя. Настройка, уничтожение или модифицирование информации
			 доступны только привилегированным пользователям.


			В менеджере безопасности настройка аудита действий пользователя
			 выполняется в разделе «[Классы
			 объектов](../04_SecurityPolicy/Admin_Object_Classes.htm)». Для получения подробной информации обратитесь
			 к разделу «[Настройка
			 аудита действий пользователя](../04_SecurityPolicy/Admin_Object_Classes.htm)». Протоколирование
			 действий пользователей осуществляется в разделе «[Протокол
			 доступа](../05_AccessProtocol/Admin_AccessProtocol.htm)». Для получения подробной информации обратитесь

			 доступа](../05_AccessProtocol/Admin_AccessProtocol.htm)».

			 проверки паролей пользователей](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General_Parole.htm)».

			 УПД.2 Реализация необходимых методов
			 управления доступом и правил разграничения доступа
			 В платформе для управления доступом субъектов доступа
			 к объектам доступа должны быть реализованы методы управления
			 доступом. Методы управления доступом реализуются в зависимости
			 от особенностей функционирования информационной системы, с
			 учетом угроз безопасности информации и должны включать один
			 или комбинацию следующих методов:


				- дискреционный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе идентификационной информации субъекта и для
				 каждого объекта доступа – списка, содержащего набор субъектов
				 доступа (групп субъектов) и ассоциированных с ними типов
				 доступа;


				- мандатный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе сопоставления классификационных меток каждого
				 субъекта доступа и каждого объекта доступа, отражающих
				 классификационные уровни субъектов доступа и объектов
				 доступа, являющиеся комбинациями иерархических и неиерархических
				 категорий;


				- метод разграничения доступа по уровням осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе сопоставления меток критичности каждого субъекта
				 доступа и каждого объекта доступа, отражающих иерархический
				 уровень безопасности;


				- атрибутный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе политик и правил, которые содержат набор условий
				 для проверки атрибутов субъектов и объектов/сегментов
				 данных.


			 В менеджере безопасности:


				- для разграничения доступа с помощью дискреционного
				 метода обратитесь к разделу «[Настройка
				 дискреционного метода](../04_SecurityPolicy/Admin_PermSep_D.htm)»;


				- для разграничения доступа с помощью мандатного метода
				 обратитесь к разделу «[Настройка
				 мандатного метода](../04_SecurityPolicy/Admin_PermSep_M.htm)»;


				- для разграничения доступа с помощью уровней безопасности
				 обратитесь к разделу «[Настройка
				 метода разграничения доступа по уровням](../04_SecurityPolicy/Security_levels.htm)»;


				- для разграничения доступа с помощью атрибутного

				 атрибутного метода](../04_SecurityPolicy/Admin_ABAC.htm)».

			 доступа](../04_SecurityPolicy/Admin_SecPolicy.htm#access_control)». Для получения подробной информации обратитесь
			 к разделу «[Выбор
			 методов разграничения доступа и их настройка](../04_SecurityPolicy/Admin_SecPolicy.htm)».

			 Права назначаются в окне «[Параметры управления
			 доступом](../03_Admin/Admin_AdminObjects.htm)» при использовании дискреционного, мандатного
			 методов разграничения доступа и разграничения доступа по уровням
			 для конкретного объекта в разделе «[Навигатор](../02_ComonFunction/Admin_SecManager_CommonFunction.htm)»
			 или для типа объектов в разделе «[Классы
			 объектов](../04_SecurityPolicy/Admin_Object_Classes.htm)». Права доступа при атрибутном методе назначаются
			 в виде правил в разделе «[Атрибутный
			 доступ](../04_SecurityPolicy/Admin_AttributeAccess.htm)». Для получения подробной информации обратитесь
			 к разделам «[Настройка
			 прав доступа к объектам](../02_ComonFunction/Admin_SecManager_CommonFunction.htm)», «[Настройка
			 аудита действий пользователя](../04_SecurityPolicy/Admin_Object_Classes.htm)», «[Добавление
			 правил и политик проверки доступа](../04_SecurityPolicy/Admin_AttributeAccess.htm)».


			Привилегии распределяются между пользователями системы в
			 разделе «[Привилегии](../04_SecurityPolicy/Admin_Priv.htm)».
			 Встроенные группы (администраторы, пользователи) по умолчанию
			 имеют разный набор привилегий.


			В платформе доступна возможность [разделения
			 ролей администраторов](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_Adm.htm) на администратора информационной

			 ролей администраторов](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_Adm.htm)».


			Вход в систему стандартным образом будет осуществляться
			 только под именем прикладного администратора, под именем администратора
			 информационной безопасности зайти в систему нельзя. Рабочим
			 местом администратора информационной безопасности является
			 менеджер безопасности, запущенный в виде отдельного приложения.
			 У администраторов будет разный набор [привилегий](../04_SecurityPolicy/Admin_Priv.htm).
			 Для получения подробной информации обратитесь к разделу «[Распределение привилегий](../04_SecurityPolicy/Admin_Priv.htm)».

			 должностным лицом, а также обеспечиваться блокирование устройства,

			 количества неуспешных попыток входа в информационную систему
			 (доступа к информационной системе).


			Ограничение количества неуспешных попыток входа в информационную
			 систему (доступа к информационной системе) должно обеспечиваться

			 В менеджере безопасности установите флажок «[Блокировать

			 РСБ.1 Определение событий безопасности, подлежащих

			 В менеджере безопасности протоколируются действия пользователей

			 аудита действий пользователя](../04_SecurityPolicy/Admin_Object_Classes.htm)».


			 РСБ.2 Определение состава и содержания информации
			 о событиях безопасности, подлежащих регистрации
			 В платформе должны быть определены состав и содержание
			 информации о событиях безопасности, подлежащих регистрации.


			Состав и содержание информации о событиях безопасности,
			 включаемой в записи регистрации о событиях безопасности, должны,
			 как минимум, обеспечить возможность идентификации типа события
			 безопасности, даты и времени события безопасности, идентификационной
			 информации источника события безопасности, результат события
			 безопасности (успешно или неуспешно), субъект доступа (пользователь
			 и (или) процесс), связанный с данным событием безопасности.


			 В менеджере безопасности протоколируются действия пользователей
			 в разделе «[Протокол
			 доступа](../05_AccessProtocol/Admin_AccessProtocol.htm)». Для получения подробной информации обратитесь
			 к разделу «[Протокол
			 доступа](../05_AccessProtocol/Admin_AccessProtocol.htm)».


			 РСБ.3 Сбор, запись и хранение информации о
			 событиях безопасности в течение установленного времени хранения
			 В платформе должны осуществляться сбор, запись и хранение
			 информации о событиях безопасности в течение установленного
			 должностным лицом времени хранения информации о событиях безопасности.


			Объем памяти для хранения информации о событиях безопасности
			 должен быть рассчитан и выделен с учетом типов событий безопасности,
			 подлежащих регистрации в соответствии с РСБ.1, составом и
			 содержанием информации о событиях безопасности, подлежащих
			 регистрации, в соответствии с РСБ.2, прогнозируемой частоты
			 возникновения подлежащих регистрации событий безопасности,
			 срока хранения информации о зарегистрированных событиях безопасности
			 в соответствии с РСБ.1.


			 Объем памяти для хранения информации о событиях безопасности
			 зависит только от ограничений используемой СУБД.


			 РСБ.4 Реагирование на сбой при регистрации
			 событий безопасности
			 В информационной системе должно осуществляться реагирование
			 на сбои при регистрации событий безопасности, в том числе
			 аппаратные и программные ошибки, сбои в механизмах сбора информации
			 и достижение предела или переполнения объема (емкости) памяти.


			Реагирование на сбои при регистрации событий безопасности
			 должно предусматривать:


				- предупреждение (сигнализация, индикация) администраторов
				 о сбоях (аппаратных и программных ошибках, сбоях в механизмах
				 сбора информации или переполнения объема (емкости) памяти)
				 при регистрации событий безопасности;


				- реагирование на сбои при регистрации событий безопасности
				 путем изменения администраторами параметров сбора, записи
				 и хранения информации о событиях безопасности, в том числе
				 отключение записи информации о событиях безопасности от
				 части компонентов информационной системы, запись поверх
				 устаревших хранимых записей событий безопасности.


			Правила и процедуры реагирования на сбои при регистрации
			 событий безопасности регламентируются в организационно-распорядительных
			 документах оператора по защите информации.


			 В менеджере безопасности для предупреждения (сигнализация,
			 индикация) администраторов о сбоях при регистрации событий
			 безопасности используются пороговые значения в разделе «[Редактор политик](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm)». Установите
			 флажок « [Пороговые
			 значения объема протокола доступа ](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm#threshold_values)» на вкладке
			 « [Общая
			 политика ](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm)» и задайте первое и второе пороговые
			 значения (число записей). При каждом входе в менеджер безопасности
			 под пользователем, имеющим привилегию «[
			 Очистка протокола доступа ](../04_SecurityPolicy/Admin_Priv.htm)», проверяется, достиг ли объем
			 протокола доступа пороговых значений. В случае, если было
			 достигнуто пороговое значение, то будет предложено очистить
			 протокол доступа. Для получения подробной информации обратитесь
			 к разделу «[
			 Общие настройки политики ](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm)».


			Для протоколирования административных операций настройте
			 объем ведения аудита в разделе «[Классы
			 объектов](../04_SecurityPolicy/Admin_Object_Classes.htm)». Для получения подробной информации обратитесь
			 к разделу «[Настройка
			 аудита действий пользователя](../04_SecurityPolicy/Admin_Object_Classes.htm)».


			 РСБ.5 Мониторинг результатов регистрации событий
			 безопасности
			 В платформе должна быть обеспечена возможность осуществлять
			 мониторинг (просмотр, анализ) результатов регистрации событий
			 безопасности и реагирование на них.


			Мониторинг (просмотр и анализ) записей регистрации (аудита)
			 должен проводиться для всех событий, подлежащих регистрации
			 в соответствии с РСБ.1, и с периодичностью, установленной
			 должностным лицом, и обеспечивающей своевременное выявление
			 признаков инцидентов безопасности в информационной системе.


			 В менеджере безопасности мониторинг нарушений защиты
			 выполняется в разделе «[Мониторинг
			 нарушений защиты](../06_MonitoringViolations/Admin_MonitoringViolations.htm)». Для получения подробной информации
			 обратитесь к разделу «[Мониторинг
			 нарушений защиты](../06_MonitoringViolations/Admin_MonitoringViolations.htm)».


			 РСБ.7 Защита информации о событиях безопасности
			 В платформе должна обеспечиваться защита информации
			 о событиях безопасности.


			Защита информации о событиях безопасности (записях регистрации
			 (аудита)) обеспечивается применением мер защиты информации
			 от неправомерного доступа, уничтожения или модифицирования,
			 определенных в соответствии с настоящим методическим документом,
			 и в том числе включает защиту средств ведения регистрации
			 (аудита) и настроек механизмов регистрации событий.


			Доступ к записям аудита и функциям управления механизмами
			 регистрации (аудита) должен предоставляться только уполномоченным
			 должностным лицам.


			 Защита информации о событиях безопасности обеспечивается
			 определенными [привилегиями](../04_SecurityPolicy/Admin_Priv.htm)
			 пользователя. Настройка, уничтожение или модифицирование информации
			 доступны только привилегированным пользователям.


			В менеджере безопасности настройка аудита действий пользователя
			 выполняется в разделе «[Классы
			 объектов](../04_SecurityPolicy/Admin_Object_Classes.htm)». Для получения подробной информации обратитесь
			 к разделу «[Настройка
			 аудита действий пользователя](../04_SecurityPolicy/Admin_Object_Classes.htm)». Протоколирование
			 действий пользователей осуществляется в разделе «[Протокол
			 доступа](../05_AccessProtocol/Admin_AccessProtocol.htm)». Для получения подробной информации обратитесь
			 к разделу «[Протокол
			 доступа](../05_AccessProtocol/Admin_AccessProtocol.htm)».


			 РСБ.8 Обеспечение возможности просмотра и
			 анализа информации о действиях отдельных пользователей в информационной
			 системе
			 В информационной системе должна иметься возможность
			 просмотра и анализа информации о действиях отдельных пользователей

			Правила и процедуры просмотра и анализа информации о действиях
			 отдельных пользователей регламентируются в организационно-распорядительных
			 документах оператора по защите информации.


			 Требование является необязательным для выполнения.

				 идентифицирует пользователя и (или) устройство;


				- присвоения идентификатора пользователю;


				- предотвращения повторного использования идентификатора

			 группу администраторов](../03_Admin/Groups/Admin_Groups.htm). При необходимости могут быть скорректированы

			 учетных записей пользователей и работа с ними](../03_Admin/Users/Admin_Users.htm)».

			 пользователей установите флажок «[Заблокировать
			 учетную запись пользователя, если пользователь неактивен](../03_Admin/Access_control_settings.htm)»

			 ИАФ.4 Управление средствами аутентификации,
			 в том числе хранение, выдача, инициализация, блокирование

			 Платформа должна обеспечивать установление характеристик

				 паролей);

				 днях);

			 УПД.2 Реализация необходимых методов
			 управления доступом и правил разграничения доступа
			 В платформе для управления доступом субъектов доступа
			 к объектам доступа должны быть реализованы методы управления
			 доступом. Методы управления доступом реализуются в зависимости
			 от особенностей функционирования информационной системы, с
			 учетом угроз безопасности информации и должны включать один
			 или комбинацию следующих методов:


				- дискреционный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе идентификационной информации субъекта и для
				 каждого объекта доступа – списка, содержащего набор субъектов
				 доступа (групп субъектов) и ассоциированных с ними типов
				 доступа;


				- мандатный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе сопоставления классификационных меток каждого
				 субъекта доступа и каждого объекта доступа, отражающих
				 классификационные уровни субъектов доступа и объектов
				 доступа, являющиеся комбинациями иерархических и неиерархических
				 категорий;


				- метод разграничения доступа по уровням осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе сопоставления меток критичности каждого субъекта
				 доступа и каждого объекта доступа, отражающих иерархический
				 уровень безопасности;


				- атрибутный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе политик и правил, которые содержат набор условий
				 для проверки атрибутов субъектов и объектов/сегментов
				 данных.

				- для разграничения доступа с помощью дискреционного
				 метода обратитесь к разделу «[Настройка
				 дискреционного метода](../04_SecurityPolicy/Admin_PermSep_D.htm)»;


				- для разграничения доступа с помощью мандатного метода
				 обратитесь к разделу «[Настройка
				 мандатного метода](../04_SecurityPolicy/Admin_PermSep_M.htm)»;

				 метода разграничения доступа по уровням](../04_SecurityPolicy/Security_levels.htm)»;


				- для разграничения доступа с помощью атрибутного

				 атрибутного метода](../04_SecurityPolicy/Admin_ABAC.htm)».


			Комбинация методов разграничения доступа выполняется в редакторе
			 политик менеджера безопасности на вкладке «[Контроль
			 доступа](../04_SecurityPolicy/Admin_SecPolicy.htm#access_control)». Для получения подробной информации обратитесь

			 методов разграничения доступа и их настройка](../04_SecurityPolicy/Admin_SecPolicy.htm)».

			 УПД.4 Разделение полномочий пользователей,

			 РСБ.4 Реагирование на сбой при
			 регистрации событий безопасности
			 В информационной системе должно осуществляться реагирование
			 на сбои при регистрации событий безопасности, в том числе
			 аппаратные и программные ошибки, сбои в механизмах сбора информации
			 и достижение предела или переполнения объема (емкости) памяти.


			Реагирование на сбои при регистрации событий безопасности
			 должно предусматривать:


				- предупреждение (сигнализация, индикация) администраторов
				 о сбоях (аппаратных и программных ошибках, сбоях в механизмах
				 сбора информации или переполнения объема (емкости) памяти)
				 при регистрации событий безопасности;


				- реагирование на сбои при регистрации событий безопасности
				 путем изменения администраторами параметров сбора, записи
				 и хранения информации о событиях безопасности, в том числе
				 отключение записи информации о событиях безопасности от
				 части компонентов информационной системы, запись поверх
				 устаревших хранимых записей событий безопасности.


			Правила и процедуры реагирования на сбои при регистрации
			 событий безопасности регламентируются в организационно-распорядительных
			 документах оператора по защите информации.

			 настройки политики](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm)».

			 безопасности
			 В платформе должна обеспечиваться защита информации
			 о событиях безопасности.


			Защита информации о событиях безопасности (записях регистрации
			 (аудита)) обеспечивается применением мер защиты информации
			 от неправомерного доступа, уничтожения или модифицирования,
			 определенных в соответствии с настоящим методическим документом,
			 и в том числе включает защиту средств ведения регистрации
			 (аудита) и настроек механизмов регистрации событий.


			Доступ к записям аудита и функциям управления механизмами

			В ОС Linux проверка контрольных сумм выполняется средствами

			 контрольных сумм в соответствии с документацией на ОС Linux.


			 Требования к усилению 1:

			 УПД.2 Реализация необходимых методов
			 управления доступом и правил разграничения доступа
			 В платформе для управления доступом субъектов доступа
			 к объектам доступа должны быть реализованы методы управления
			 доступом. Методы управления доступом реализуются в зависимости
			 от особенностей функционирования информационной системы, с
			 учетом угроз безопасности информации и должны включать один
			 или комбинацию следующих методов:


				- дискреционный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе идентификационной информации субъекта и для
				 каждого объекта доступа – списка, содержащего набор субъектов
				 доступа (групп субъектов) и ассоциированных с ними типов
				 доступа;


				- мандатный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе сопоставления классификационных меток каждого
				 субъекта доступа и каждого объекта доступа, отражающих
				 классификационные уровни субъектов доступа и объектов
				 доступа, являющиеся комбинациями иерархических и неиерархических
				 категорий;


				- метод разграничения доступа по уровням осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе сопоставления меток критичности каждого субъекта
				 доступа и каждого объекта доступа, отражающих иерархический
				 уровень безопасности;


				- атрибутный метод управления доступом осуществляет
				 управление доступом субъектов доступа к объектам доступа
				 на основе политик и правил, которые содержат набор условий
				 для проверки атрибутов субъектов и объектов/сегментов
				 данных.


			 В менеджере безопасности:


				- для разграничения доступа с помощью дискреционного
				 метода обратитесь к разделу «[Настройка
				 дискреционного метода](../04_SecurityPolicy/Admin_PermSep_D.htm)»;


				- для разграничения доступа с помощью мандатного метода
				 обратитесь к разделу «[Настройка
				 мандатного метода](../04_SecurityPolicy/Admin_PermSep_M.htm)»;


				- для разграничения доступа с помощью уровней безопасности
				 обратитесь к разделу «[Настройка
				 метода разграничения доступа по уровням](../04_SecurityPolicy/Security_levels.htm)»;


				- для разграничения доступа с помощью атрибутного
				 метода обратитесь к разделу «[Настройка
				 атрибутного метода](../04_SecurityPolicy/Admin_ABAC.htm)».


			Комбинация методов разграничения доступа выполняется в редакторе
			 политик менеджера безопасности на вкладке «[Контроль
			 доступа](../04_SecurityPolicy/Admin_SecPolicy.htm#access_control)». Для получения подробной информации обратитесь

			 РСБ.4 Реагирование на сбой при
			 регистрации событий безопасности
			 В информационной системе должно осуществляться реагирование
			 на сбои при регистрации событий безопасности, в том числе
			 аппаратные и программные ошибки, сбои в механизмах сбора информации
			 и достижение предела или переполнения объема (емкости) памяти.


			Реагирование на сбои при регистрации событий безопасности
			 должно предусматривать:


				- предупреждение (сигнализация, индикация) администраторов
				 о сбоях (аппаратных и программных ошибках, сбоях в механизмах
				 сбора информации или переполнения объема (емкости) памяти)
				 при регистрации событий безопасности;


				- реагирование на сбои при регистрации событий безопасности
				 путем изменения администраторами параметров сбора, записи
				 и хранения информации о событиях безопасности, в том числе
				 отключение записи информации о событиях безопасности от
				 части компонентов информационной системы, запись поверх
				 устаревших хранимых записей событий безопасности.


			Правила и процедуры реагирования на сбои при регистрации
			 событий безопасности регламентируются в организационно-распорядительных
			 документах оператора по защите информации.


			 В менеджере безопасности для предупреждения (сигнализация,

			 политика](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm)» и задайте первое и второе пороговые значения

			 настройки политики](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General.htm)».

			 безопасности
			 В платформе должна обеспечиваться защита информации
			 о событиях безопасности.


			Защита информации о событиях безопасности (записях регистрации
			 (аудита)) обеспечивается применением мер защиты информации

			 определенных в соответствии с настоящим методическим документом,

			Доступ к записям аудита и функциям управления механизмами
			 регистрации (аудита) должен предоставляться только уполномоченным

			Для разграничения прав доступа на типы объектов выберите
			 тип объектов в разделе «[Классы
			 объектов](../04_SecurityPolicy/Admin_Object_Classes.htm)» и откройте окно «[Параметры
			 управления доступом](../03_Admin/Admin_AdminObjects.htm)». На вкладке «[Дискреционный
			 контроль](../03_Admin/Admin_AdminObjects_Discretionary.htm)» добавьте субъекта безопасности (пользователя
			 или группу пользователей) с помощью окна «[Выбор
			 пользователей и групп](../03_Admin/Admin_UsersGroups.htm)», после чего субъект безопасности
			 будет добавлен в список групп и пользователей, для которых
			 могут быть разграничены права доступа. Выберите добавленного
			 субъекта безопасности в списке «Группы
			 или пользователи» и установите флажки напротив операций
			 в списке «Разрешения»,
			 которые будут разрешены/запрещены субъекту безопасности для
			 работы с типом объектов. Для получения подробной информации
			 обратитесь к разделу «[Настройка
			 аудита действий пользователя](../04_SecurityPolicy/Admin_Object_Classes.htm)».


			Привилегии распределяются между пользователями системы в
			 разделе «[Привилегии](../04_SecurityPolicy/Admin_Priv.htm)».
			 Встроенные группы (администраторы, пользователи) по умолчанию
			 имеют разный набор привилегий. В платформе доступна возможность
			 [разделения
			 ролей администраторов](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_Adm.htm) на администратора информационной
			 безопасности и прикладного администратора. Для получения подробной
			 информации обратитесь к разделу «[Разделение
			 ролей администраторов](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_Adm.htm)».


			Вход в систему стандартным образом будет осуществляться
			 только под именем прикладного администратора, под именем администратора
			 информационной безопасности зайти в систему нельзя. Рабочим
			 местом администратора информационной безопасности является
			 менеджер безопасности, запущенный в виде отдельного приложения.
			 У администраторов будет разный набор привилегий. Для получения
			 подробной информации обратитесь к разделу «[Распределение
			 привилегий](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_Adm.htm)».


			 Очистка памяти


			 При первоначальном назначении или при перераспределении
			 внешней памяти КСЗ должен предотвращать доступ субъекту к
			 остаточной информации.


			 В менеджере безопасности откройте раздел «[Редактор политик](../03_Admin/Access_control_settings.htm)»
			 и установите флажок «[Очистка
			 памяти](../03_Admin/Access_control_settings.htm#clearing_memory)». Для получения подробной информации обратитесь
			 к разделу «[Дополнительные
			 настройки контроля доступа](../03_Admin/Access_control_settings.htm)».


			 Идентификация и аутентификация


			 КСЗ должен требовать от пользователей идентифицировать
			 себя при запросах на доступ. КСЗ должен подвергать проверке
			 подлинность идентификации – осуществлять аутентификацию. КСЗ
			 должен располагать необходимыми данными для идентификации
			 и аутентификации. КСЗ должен препятствовать доступу к защищаемым
			 ресурсам неидентифицированных пользователей и пользователей,
			 подлинность идентификации которых при аутентификации не подтвердилась.


			 В «Форсайт. Аналитическая платформа»
			 доступно несколько методов аутентификации. Метод аутентификации
			 выбирается в зависимости от требуемых мер безопасности. При
			 интегрированной доменной аутентификации для доменного пользователя,
			 под которым совершен вход в операционную систему, используются
			 средства операционной системы и СУБД. При двухфакторной аутентификации
			 для пользователя используется сертификат, который позволяет
			 получить данные через обращение к СУБД. Для получения подробной
			 информации обратитесь к разделу «[Аутентификация
			 в продукте «Форсайт. Аналитическая платформа](Setup.chm::/UiWebSetup/Authentication/Authentication.htm)».


			В менеджере безопасности при создании пользователя задаются
			 учётные данные, которые используются для аутентификации и
			 идентификации. Для создания пользователя откройте раздел «[Пользователи](../03_Admin/Users/Admin_Users.htm)»
			 и задайте учётные данные с помощью окна «[Свойства пользователя](../03_Admin/Users/Admin_UserCreate.htm)»,
			 в том числе логин и пароль пользователя, определение доступа
			 с рабочих станций, IP-адресов и т.д. Для получения подробной
			 информации обратитесь к разделу «[Создание
			 и редактирование учетной записи пользователя](../03_Admin/Users/Admin_UserCreate.htm)».


			Для настройки парольной политики откройте раздел «[Редактор
			 политик](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General_Parole.htm)» и перейдите на вкладку «[Парольная
			 политика](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General_Parole.htm)». Отметьте флажками необходимые функции
			 для настройки проверки паролей пользователей и задайте им
			 значения. Для получения подробной информации обратитесь к
			 разделу «[Настройка
			 проверки паролей пользователей](../04_SecurityPolicy/Editor_of_Politicy/Security_EditorPoliticy_General_Parole.htm)».


			Аутентификация выполняется при входе в систему с помощью
			 окна регистрации, в котором указываются логин и пароль пользователя.
			 Если проверка пароля пользователя прошла успешно, то будет
			 открыт репозиторий в навигаторе объектов.


			 Регистрация


			 КСЗ должен быть в состоянии осуществлять регистрацию
			 следующих событий:


				- использование идентификационного и аутентификационного
				 механизма;


				- запрос на доступ к защищаемому ресурсу (открытие
				 файла, запуск программы и т.д.);


				- создание и уничтожение объекта;


				- действия по изменению ПРД.


			Для каждого из этих событий должна регистрироваться следующая
			 информация:


				- дата и время;


				- субъект, осуществляющий регистрируемое действие;


				- тип события (если регистрируется запрос на доступ,
				 то следует отмечать объект и тип доступа);


				- успешно ли осуществилось событие (обслужен запрос
				 на доступ или нет).


			КСЗ должен содержать средства выборочного ознакомления с
			 регистрационной информацией.


			 В менеджере безопасности протоколируются действия пользователей
			 в разделе «[Протокол доступа](../05_AccessProtocol/Admin_AccessProtocol.htm)».
			 Для получения подробной информации обратитесь к разделу «[Протокол
			 доступа](../05_AccessProtocol/Admin_AccessProtocol.htm)».


			Для протоколирования операций для каждого типа объектов
			 в менеджере безопасности должен быть настроен аудит в разделе
			 «[Классы
			 объектов](../04_SecurityPolicy/Admin_Object_Classes.htm)». Для получения подробной информации обратитесь
			 к разделу «[Настройка
			 аудита действий пользователя](../04_SecurityPolicy/Admin_Object_Classes.htm)».


			 Целостность КСЗ


			 В СВТ пятого класса защищенности должны быть предусмотрены
			 средства периодического контроля за целостностью программной
			 и информационной части КСЗ.


			 При установке продукта должен быть установлен флажок
			 «[Проверять
			 контрольные суммы при каждом запуске платформы](Setup.chm::/02_AK_Install/Setup_AK_Install.htm#sum)».
			 При установленном флажке при каждом запуске «Форсайт. Аналитическая платформа»
			 будет проверяться [целостность](../04_SecurityPolicy/Integrity_control.htm)
			 исполняемых файлов. Проверка контрольных сумм для исполняемых
			 файлов производится в момент их загрузки. При каждом запуске
			 «Форсайт. Аналитическая платформа»
			 загружается минимальный набор исполняемых файлов, а по мере
			 обращения к функциональности загружаются дополнительные библиотеки,
			 необходимые для работы.


			В ОС Linux проверка контрольных сумм выполняется средствами
			 операционной системы. При необходимости настройте проверку
			 контрольных сумм в соответствии с документацией на ОС Linux.


			Для проверки целостности метаданных репозитория используется
			 [расчёт
			 контрольных сумм](../04_SecurityPolicy/Integrity_control.htm#performing_checksum), которые вычисляются определенным образом
			 по содержимому объекта. Для получения подробной информации
			 обратитесь к подразделу «[Проверка
			 целостности метаданных репозитория](../04_SecurityPolicy/Integrity_control.htm#metadata)».
